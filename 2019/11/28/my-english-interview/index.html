你们开发周期是怎样的 上下游如何协同

What is your development cycle?
How upstream and downstream work together?

通常我们每周四发布一个版本，周一至周三开发新功能，周四评估是否能上线，如果线上出了bug则会立即修复并发布。

我们使用worktile工具进行协同合作，产品经理负责发布任务，开发人员承接任务，开发结束后交由测试部门测试，通过后运维部署生产环境。

Usually we release a version every Thursday, develop new features from Monday to Wednesday, and evaluate whether it can go online on Thursday. If there is a bug online, it will be fixed and released immediately.

We use the worktile tool for collaboration. The product manager is responsible for publishing tasks, and the developers undertake the tasks. After the development, they are tested by the testing department, and the production environment is deployed after the operation and maintenance.

有没有在生产环境处理定位过问题 如何做的

Have you dealt with positioning problems in the production environment?

为了能够追踪生产的服务日志，我们并没有接入分布式追踪工具，如Zipkin，而是自己写了一个很简单的过滤器，只传递一个元素 traceId,实现方式是通过logback MDC功能传递，但是某次版本发布后，发现子线程中traceId不见了，最后通过比对war包内部lib文件夹下的jar才发现是logback的版本被修改了，新版本中MDC对于子线程的实现去除了inheritablethreadlocal，我们项目的pom文件都是继承superpom文件，有人修改了logback版本

In order to be able to track production service logs, we did not access distributed tracking tools such as Zipkin, but wrote a very simple filter ourselves, passing only one element traceId. The implementation method is passed through the logback MDC function. After the release of a version, it was found that the traceId was missing in the child thread. Finally, it was found that the version of logback was modified by comparing the jars in the lib folder inside the war package. In the new version, the implementation of child threads by MDC removed inheritablethreadlocal. The pom files of the project are inherited from the superpom file. Someone modified the logback version.

如何设计一个幂等的接口

How to design an idempotent interface?

幂等操作的特点是任意多次执行和和一次执行的影响是一样的。
对于每个请求必须有一个唯一的标识，例如，订单支付请求，肯定得包含订单 id，一个订单 id 最多支付一次
要求是支付一个订单，必须插入一条支付流水，order_id 建一个唯一键 unique key。在支付一个订单之前，先插入一条支付流水，order_id 就已经进去了

The characteristic of idempotent operation is that the execution of any number of executions has the same effect as one execution.
For each request, there must be a unique identifier. For example, an order payment request must include the order id. An order id must be paid at most once.
The requirement is to pay for an order, a payment pipeline must be inserted, and order_id builds a unique key. Before paying an order, insert a payment flow, and order_id is already entered.
